Análise e Modelagem de Sistemas

Guia cronológico de análise e modelagem — do problema ao sistema documentado

01

Entendimento do Problema e Contexto

Antes de qualquer diagrama ou documento, você precisa entender o negócio. Essa fase é sobre escutar, questionar e registrar. Sem ela, tudo que vier depois pode estar errado desde o início.

O que fazer nessa etapa
Entrevistas com usuários e gestoresPergunte: "Como funciona hoje?", "O que não funciona bem?", "O que você gostaria de melhorar?"
Registro de problemas e oportunidadesAnote os "pontos de dor" relatados — esses viram requisitos depois
Entendimento do negócioQual o setor? Qual o objetivo do sistema? Quem são os clientes?
Exemplo prático
Situação: Uma farmácia quer um sistema de controle de estoque.

Antes de modelar qualquer coisa, você entrevista o farmacêutico e descobre: "Todo mês eu perco medicamentos vencidos porque não sei o que está prestes a vencer" e "Às vezes vendo um produto que acabou porque o sistema não atualiza na hora."

Esses dois problemas já orientam boa parte dos requisitos que virão.
⚡ Dica: Use a técnica dos 5 Porquês — quando alguém citar um problema, pergunte "por quê?" cinco vezes. Você vai chegar na causa raiz, não no sintoma.
02

Levantamento de Stakeholders e Matriz RACI

Stakeholders são todas as pessoas que afetam ou são afetadas pelo sistema. Identificá-los cedo evita surpresas tardias e conflitos de requisitos.

Quem são os stakeholders?
Usuários diretos
Quem vai usar o sistema no dia a dia (ex: atendente, caixa, operador)
Gestores
Quem toma decisões e aprova o projeto (ex: gerente, diretor)
TI / Técnicos
Equipe que vai desenvolver e manter o sistema
Externos
Fornecedores, parceiros, clientes, órgãos reguladores
Matriz RACI — o que entregar
RACI = Responsável · Aprovador · Consultado · Informado

Atividade Analista Gerente Dev Usuário
Levantar requisitos R A C C
Validar protótipo R A I C
⚡ Dica: A Matriz RACI evita o clássico "achei que era você que ia fazer isso". Defina no início quem aprova cada entrega.
03

Mapeamento do Processo AS IS (situação atual)

Documentar como o processo funciona hoje — com todos os seus problemas e gambiarras. Isso serve como linha de base para propor melhorias depois.

Ferramentas e notações
BPMN
Business Process Model and Notation — padrão internacional para mapear processos com raias, eventos, atividades e gateways
DFD
Diagrama de Fluxo de Dados — mostra como os dados fluem entre processos, entidades externas e depósitos
Fluxograma simples
Mais informal, bom para esboços rápidos em entrevistas. Depois formalizar em BPMN
Elementos-chave do BPMN para lembrar
Eventos (círculos)Início (borda simples), Intermediário (borda dupla), Fim (borda grossa). Sempre há um evento de início e pelo menos um de fim.
Atividades / Tarefas (retângulos arredondados)O que alguém ou o sistema faz. Use verbos no infinitivo: "Validar CPF", "Emitir NF"
Gateways (losangos)Pontos de decisão ou paralelismo. Gateway Exclusivo (X) = só um caminho. Paralelo (+) = todos os caminhos.
Piscinas e raias (Pools e Lanes)Piscina = organização (ex: Empresa). Raias dentro da piscina = departamentos ou papéis (ex: Vendas, Financeiro).
O que entregar ao final desta fase
Diagrama AS IS em BPMN com raias por responsável, descrevendo o processo atual completo, incluindo pontos de decisão, esperas e exceções. Ferramentas sugeridas: Bizagi Modeler (gratuito), Lucidchart, draw.io.

Observações de problemas identificados no processo: gargalos, retrabalho, atividades manuais que poderiam ser automatizadas, etapas sem responsável claro.
04

Engenharia de Requisitos

Requisitos são o coração do projeto. São eles que dizem o que o sistema deve fazer (e como deve se comportar). Mal levantados = projeto fracassado.

Tipos de requisitos
Funcionais (RF)
O que o sistema deve fazer. Funcionalidades, comportamentos, regras de negócio.

Ex: "O sistema deve permitir cadastrar medicamentos com data de validade."
Não Funcionais (RNF)
Como o sistema deve se comportar. Performance, segurança, usabilidade, disponibilidade.

Ex: "O sistema deve responder em menos de 2 segundos."
Como documentar um requisito (exemplo)
RF-001 — Controle de validade
Descrição: O sistema deve alertar o farmacêutico quando um medicamento atingir 30 dias antes do vencimento.
Prioridade: Alta
Origem: Entrevista com farmacêutico (03/06/2025)
Critério de aceite: Sistema exibe notificação na tela inicial com nome, quantidade e data de vencimento dos itens críticos.

RNF-001 — Disponibilidade
Descrição: O sistema deve estar disponível 99,5% do tempo em horário comercial (8h–20h).
Critério de aceite: Máximo de 22 minutos de indisponibilidade por mês nesses horários.
⚡ Todo requisito precisa de um critério de aceite — uma frase que diz "quando isso está feito e aprovado". Sem ele, ninguém sabe quando termina.
Técnicas de levantamento de requisitos
Entrevistas1 a 1 com stakeholders — mais rico, porém demorado
ObservaçãoAcompanhe o usuário no trabalho real — descobre coisas que ele não conta em entrevista
Análise de documentosFormulários, relatórios, planilhas existentes revelam dados e regras implícitas
PrototipaçãoMostrar uma tela mesmo que simples — o usuário entende melhor o que quer ao ver algo concreto
05

Modelagem do Processo TO BE (situação futura)

Com o AS IS mapeado e os requisitos levantados, agora você projeta como o processo vai funcionar com o novo sistema — mais eficiente, sem os problemas identificados.

Diferença entre AS IS e TO BE
AS IS — Processo Atual
Descreve a realidade de hoje, com todos os problemas. Serve para entender e identificar melhorias.
TO BE — Processo Futuro
Descreve como o processo vai funcionar após a solução. Etapas manuais viram automáticas, fluxos são simplificados.
Exemplo de transformação AS IS → TO BE
Processo: Controle de estoque de medicamentos

AS IS: Farmacêutico verifica planilha manualmente toda semana → conta estoque físico → anota diferenças no caderno → avisa fornecedor por telefone → espera confirmação por e-mail.

TO BE: Sistema atualiza estoque em tempo real a cada venda → alerta automático quando item atinge estoque mínimo → pedido de compra gerado automaticamente → fornecedor recebe pedido via integração e confirma no sistema.
⚡ Ao construir o TO BE em BPMN, use as mesmas raias do AS IS. Assim fica fácil comparar e mostrar para os stakeholders o que mudou.
06

Modelagem UML e Diagramas do Sistema

UML (Unified Modeling Language) é a linguagem padrão para documentar sistemas orientados a objetos. Você vai criar diferentes visões do sistema usando diagramas específicos.

Diagrama de Caso de Uso — o primeiro a fazer
Objetivo: Mostrar o que o sistema faz, do ponto de vista de quem usa.

Elementos:
🧑 Ator = quem interage com o sistema (usuário, sistema externo)
⭕ Caso de uso = funcionalidade do sistema (elipse com nome no infinitivo)
— Associação = linha ligando ator ao caso de uso
«include» = um caso de uso que sempre inclui outro
«extend» = um caso de uso que pode incluir outro opcionalmente

Exemplo: Ator "Farmacêutico" → "Consultar estoque", "Registrar venda", "Emitir alerta de vencimento"
Outros diagramas UML importantes
Diagrama de ClassesEstrutura do sistema OO: classes, atributos, métodos e relacionamentos (herança, composição, associação). Base para o desenvolvimento.
Diagrama de SequênciaComo os objetos interagem ao longo do tempo para realizar um caso de uso. Mostra troca de mensagens com linha do tempo.
Diagrama de EstadosEstados que um objeto pode assumir e as transições entre eles. Ex: Pedido pode ser (Aberto → Em processamento → Enviado → Entregue → Cancelado).
Diagrama de AtividadesParecido com fluxograma, mas com notação UML. Bom para detalhar a lógica de um caso de uso complexo.
⚡ Não é necessário fazer todos os diagramas — escolha os que agregam valor para comunicar o sistema. Diagrama de Caso de Uso + Classes são os mais usados na prática.
Documentando casos de uso em detalhe
UC-001: Registrar Venda
Ator principal: Operador de caixa
Pré-condição: Operador autenticado no sistema
Fluxo principal:
1. Operador escaneia código de barras do produto
2. Sistema exibe nome, preço e estoque disponível
3. Operador confirma quantidade
4. Sistema atualiza estoque e gera comprovante
Fluxo alternativo: Se produto não encontrado → sistema exibe mensagem de erro
Pós-condição: Estoque atualizado; venda registrada no histórico
07

Gestão de Riscos, Critérios de Aceite e Estratégia de Testes

Antes de encerrar a análise, você precisa antecipar o que pode dar errado, definir como será validado e planejar os testes.

Gestão de riscos — tabela básica
Risco Probab. Impacto Ação
Requisitos mal levantados Alta Alto Validar com cliente em ciclos curtos
Stakeholder indisponível Média Médio Definir substituto na Matriz RACI
Escopo crescente (scope creep) Alta Médio Congela escopo por escrito aprovado
Critérios de aceite e estratégia de testes
Critério de aceiteCada requisito precisa de um critério objetivo. Ex: RF-001 está aceito quando o sistema exibe alerta em menos de 5 segundos após detectar produto com validade ≤ 30 dias.
Testes funcionaisVerificam se o sistema faz o que o requisito manda. Cada RF vira pelo menos um caso de teste.
Testes não funcionaisVerificam performance, carga, segurança. Ex: Simular 100 usuários simultâneos e medir tempo de resposta.
Teste de aceite do usuário (UAT)O próprio usuário executa cenários reais para validar se o sistema atende suas necessidades. Esse é o teste final.
